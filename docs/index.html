The problem

AI coding agents got good enough to trust with real work, and that is exactly the problem. The moment an agent can edit files, run shell commands, and push migrations, “let it run and see” stops being a strategy — so you babysit a terminal, and everyone else on the team gets either no access to the agent or an access level that terrifies you.

bow-agent runs on your machine and puts one permission boundary between the agent and anything that writes. Your QC engineer, your BA, your tech lead, and your contractor all work through it — each with only the reach their job needs.

Six modes, one agent

Same core. Same gate. Six different amounts of rope.

ModeCommandCan doCannot do
Devnpm run ui Everything — with an approval card for every write —
QCnpm run ui:qc:share Read source, triage Jira tickets, comment & transition issues Touch a single line of code
Collabnpm run ui:collab Write code, run tests Write anything — including git — without your live approval
BAnpm run ui:ba Write docs and *.md, full Jira access Source, database, deploy — hard-denied
Reviewernpm run ui:review:share Review PRs, comment and approve on GitHub, run tests Edit code, merge, push
DevOpsnpm run ui:devops:share Write infra — Dockerfile, compose, workflows, Terraform, k8s/Helm Touch app source; deploy needs your approval

Each mode runs on its own port, so they all run at once. Admin is the machine at the real socket address — a LAN client cannot forge a header to seize it.

What you get

ONE GATE

No path around it

Reads run free. Every write — edits, side-effecting shell, Jira comments, SQL — routes through a single function, canUseTool. Auditing this project means reading one file.

REMOTE APPROVAL

Their agent, your keyboard

A contractor’s agent wants to write? The approval card pops up on your screen, with the diff, and waits. Nothing lands on disk until you click.

TICKET → PLAN

Paste a Jira URL

It pulls the ticket, the acceptance criteria, the screenshots — and the video attachments, which it watches. Then it plans, and touches nothing until you approve.

YOUR REPO

Not a generic codebase

Point it anywhere with --cwd. It reads that repo’s CLAUDE.md, detects the stack, and can write a lasting knowledge profile for a codebase it has never seen.

SUBAGENTS

A reviewer that argues back

Opt in and the main agent delegates: a reviewer that attacks the plan, a verifier that traces runtime behaviour, a scout that finds every call site before you change a contract. All read-only.

SELF-HOSTED

Runs on your machine

Your code never leaves your network. Uses your existing Claude subscription through the CLI — no API key, no separate bill, no third-party server in the middle.

Contribute

Ideas count as contributions.

This started as an internal tool and went open because the problem turned out not to be internal at all. If you have fought “how do I let my team use an AI agent without giving them root” — I want to hear how you solved it. No code required.

More roles There are five. What about a designer? A PM? A security reviewer? Each one is a permission policy — the hard part is deciding what the role should never touch.
More stacks The base profile assumes Flutter + Supabase because that is what I build. Rails, Django, Go, Next.js — encode your stack’s conventions and it works for people like you.
Auth beyond LAN Sharing means “same network, admin approves you by name.” Real auth would let teams use this remotely.
